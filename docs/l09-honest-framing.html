ENCRYPTION · LESSON 01

Encryption with XSLT — the honest framing

XSLT is a transformer, not a cipher engine. Here are the three real paths, and the rules that keep you safe.

“Can you do encryption with XSLT?” is the question that brought many people to this page. The honest answer has two halves, and you need both: yes, you can build ciphers in XSLT — and no, you must not use them to protect anything real. This lesson draws that line clearly, then lays out the three legitimate paths the rest of the track follows.

XSLT is a transformer, not a cipher engine

XSLT was designed to reshape documents. It has no random-number generator, no bitwise operators, no timing guarantees, and no access to your OS key store — the very things real cryptography depends on. You can express a cipher's arithmetic in XSLT, the same way you can express one in a spreadsheet. That makes it a wonderful teaching tool and a terrible security tool.

⚠ The rule, stated once and for all

Never protect real data with a cipher you implemented yourself — in XSLT or any other language. Real confidentiality comes from vetted libraries (AES, RSA, ChaCha20) driven by careful key management. XSLT's honest role is to decide what to protect and where it goes; a real crypto library does the how.

First, get the words right

Half of all “encryption” confusion is vocabulary. These are not synonyms:

Base64 is the one people misfile most often. It is encoding — it protects nothing. Anyone can decode it in one command. We build it in the next lesson to learn byte math, not to hide anything.

The three honest paths

Path 1 · Learn

Teaching ciphers in pure XSLT. Caesar, Base64, XOR — built by hand to master recursion and byte manipulation. Zero security value. Lesson 10 →

Path 2 · Standard

W3C XML Encryption. The real standard for encrypting parts of an XML document. XSLT selects and reassembles; a crypto library encrypts. Lesson 11 →

Path 3 · Production

Extension functions. Call AES/HMAC/hashing from a vetted library inside the transform. The pattern used in Java, .NET, and Python. Lesson 12 →

Security rules that outlive any language

  • Don't roll your own crypto. Use AES-GCM / ChaCha20-Poly1305 from a maintained library. Amateur ciphers fail in ways you can't see.
  • Assume the algorithm is public (Kerckhoffs's principle). Only the key is secret. A cipher that relies on nobody knowing how it works is already broken.
  • Authenticate, don't just encrypt. Use authenticated encryption (GCM) or add an HMAC, so tampering is detected.
  • Keys are the hard part. Never hard-code keys in a stylesheet. Derive them with a KDF (HKDF/PBKDF2) or fetch from a KMS; rotate them.
  • Mind XSLT-specific hazards: disable external entity/DTD loading to avoid XXE, never log plaintext, and remember an XSLT variable holding a secret may end up in memory dumps or error traces.

★ Practical use case — field-level protection in a message

A hospital sends XML referrals between systems. The routing header, department, and timestamps must stay readable for middleware — but the patient's name and record number must be encrypted end-to-end. You don't encrypt the whole document; you encrypt specific elements. Deciding which elements is a transformation problem (XSLT); encrypting them is a crypto-library problem. Lessons 11–12 build exactly this.

⚑ Carry this forward

For the rest of the track, watch which half of the job is XSLT's. When you see a cipher written in pure XSLT, it's a lesson. When you see real protection, XSLT is only steering — the cryptography lives in a library that experts maintain.
