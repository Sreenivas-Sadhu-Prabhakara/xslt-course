ENCRYPTION · LESSON 03

W3C XML Encryption — and XSLT's real role

Encrypt only the sensitive elements of a document. XSLT decides what and where; a vetted library does the how.

This is real encryption done the standard way. W3C XML Encryption (the xmlenc standard) lets you encrypt selected parts of an XML document while the rest stays plain and routable. XSLT plays its honest role: it decides which elements are sensitive and reassembles the result. A vetted crypto library does the actual AES. Folder: examples/11-xml-encryption.

Why element-level, not whole-file

Encrypting the entire document would blind the middleware that needs to route it. So the standard encrypts individual elements in place: an order id and totals stay readable; the card number and CVV become ciphertext.

What an encrypted element looks like

The standard replaces a sensitive element with an <EncryptedData> element in the xmlenc namespace:

the W3C EncryptedData structure
<xenc:EncryptedData xmlns:xenc="http://www.w3.org/2001/04/xmlenc#"
                    Type=".../xmlenc#Element">
  <xenc:EncryptionMethod Algorithm=".../xmlenc11#aes256-gcm"/>
  <xenc:CipherData>
    <xenc:CipherValue>svwBpoESsmsBsHat…Base64(nonce+ciphertext+tag)…</xenc:CipherValue>
  </xenc:CipherData>
</xenc:EncryptedData>

In a full deployment the symmetric key is itself wrapped with the recipient's public key and travels alongside as an <EncryptedKey> — so only the intended reader can unwrap it. Our example uses a shared demo key to keep the focus on structure.

XSLT's job: declarative policy

protect-policy.xsl is an identity transform (copy everything unchanged) with one addition: it flags the sensitive element types. The policy is one auditable place that says what counts as sensitive.

protect-policy.xsl
<!-- identity: copy every node and attribute as-is -->
<xsl:template match="@*|node()">
  <xsl:copy><xsl:apply-templates select="@*|node()"/></xsl:copy>
</xsl:template>

<!-- policy: these element types hold sensitive data → flag them -->
<xsl:template match="pan | cvv | email">
  <xsl:copy>
    <xsl:attribute name="sec:protect">true</xsl:attribute>
    <xsl:apply-templates select="@*|node()"/>
  </xsl:copy>
</xsl:template>

The crypto step (xmlenc.py, using AES-256-GCM from Python's cryptography) then acts only on flagged elements — encrypting each one and swapping in its EncryptedData.

The pipeline, end to end

$ python3 11-xml-encryption/xmlenc.py encrypt secret-order.xml
<order id="SO-2026-0042"> <customer> <name>Ada Lovelace</name> ← still readable <xenc:EncryptedData …>…</xenc:EncryptedData> ← was <email> </customer> <payment> <method>card</method> ← still readable <xenc:EncryptedData …>…</xenc:EncryptedData> ← was <pan> <xenc:EncryptedData …>…</xenc:EncryptedData> ← was <cvv> <expiry>2029-04</expiry> </payment> <total currency="USD">149.85</total> ← still readable </order>
$ python3 11-xml-encryption/xmlenc.py decrypt order.enc.xml | diff - secret-order.xml
# (no differences) — decrypt restores the original exactly. ROUND-TRIP OK

⚑ The division of labour, made concrete

XSLT: identity transform + policy (what/where), and reassembly. Library: key handling, nonce generation, AES-GCM, authentication tag. Neither could safely do the other's job — and that's the whole point.

⚠ Production notes

Use EncryptedKey with recipient public keys instead of a shared secret; derive/manage keys via a KMS; keep the authenticated-encryption tag (GCM) so tampering is caught; and disable DTD/external-entity loading on your parser to prevent XXE. Libraries like Apache Santuario (Java) and xmlsec implement the full standard including signatures.

★ Practical use case — partner data exchange & compliance

B2B and healthcare/finance exchanges routinely send documents where a few fields are regulated (PCI card data, PII, PHI) but most fields must stay visible for routing, billing, and audit. Element-level XML Encryption is the standard answer, and a policy stylesheet is the cleanest way to keep the “what is sensitive” decision in one reviewable place. The capstone puts this to work on an order feed.
