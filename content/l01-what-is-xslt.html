XSLT — eXtensible Stylesheet Language Transformations — is a language whose only job is to turn one XML document into something else: an HTML page, a text file, JSON, or a differently-shaped XML document. You write a stylesheet full of rules; a processor applies those rules to your source document and produces a result.

The mental model: two trees and a processor

An XML document isn't really text to XSLT — it's a tree of nodes (elements, attributes, text). XSLT reads the source tree and builds a brand-new result tree. Your stylesheet never modifies the source; it describes the output.

Source tree 
catalog
 └ book
    ├ title
    └ author
▸
Processor + stylesheet 
xsltproc / Saxon / lxml
applies your rules
to every node
▸
Result tree 
html
 └ body
    └ ul
       └ li ...

Declarative, not step-by-step

Most languages you know are imperative: do this, then this, then that. XSLT is mostly declarative and rule-based. Instead of writing a loop that walks the document, you write rules that say “whenever you see a node that looks like this, produce that.” The processor handles the walking.

a rule, in plain terms
When you meet a <book>, emit an <li> containing its title.
(You never say "start at the top and loop" — the processor does that.)

That flips how you think. A big part of learning XSLT is trusting the processor to visit nodes for you, and just writing good rules — you'll meet these rules properly in Lesson 4.

XSLT and XPath are a pair

XSLT leans on a second little language, XPath, to point at nodes. Every match="book" and select="title" you'll write is XPath. Think of it this way:

  • XPath answers “which nodes?” — it selects and queries.
  • XSLT answers “now what?” — it produces output for them.

We give XPath its own lesson (Lesson 3) because it does so much of the work.

★ Practical use case — one feed, many outputs

A publisher stores each article once as XML. One stylesheet renders it as a web page, another as a print-ready document, another as an RSS item, another as the JSON a mobile app expects. Author once, transform many times — nobody re-keys the content, and every channel stays in sync. That “single source, many shapes” pattern is XSLT's home turf.

⚑ Keep in mind

A stylesheet is itself a well-formed XML document. Everything you write lives inside <xsl:stylesheet>, and every instruction is an element in the xsl: namespace. XSLT is XML that produces XML (or text).
