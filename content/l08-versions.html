One stylesheet that needs 3.0

Turning the catalogue into JSON uses four things 1.0 simply doesn't have: a typed xsl:function, regex via replace, maps/arrays, and JSON serialization.

catalog-to-json.xsl (XSLT 3.0)
<xsl:output method="json" indent="yes"/>

<xsl:function name="c:slug" as="xs:string">
  <xsl:param name="s" as="xs:string"/>
  <xsl:sequence select="lower-case(replace(normalize-space($s), '\s+', '-'))"/>
</xsl:function>

<xsl:template match="/catalog">
  <xsl:sequence select="array {
      for $b in book return map {
        'id'    : string($b/@id),
        'slug'  : c:slug($b/title),
        'title' : string($b/title),
        'price' : xs:decimal($b/price),
        'tags'  : array { $b/tags/tag ! string(.) }
      }
    }"/>
</xsl:template>
$ java -jar lib/Saxon-HE-10.9.jar -xsl:07-versions/catalog-to-json.xsl -s:data/catalog.xml
[ { "price": 39.95, "slug": "learning-xslt", "title": "Learning XSLT", "tags": [ "xml","xslt" ], "id": "b1" }, … one object per book … ]

ⓘ the ||, ! and for you just saw

|| concatenates strings; ! is the “simple map” (apply an expression to each item); for $x in … return … loops as an expression. These make 2.0/3.0 feel like a real functional language rather than a template engine.

Which version should you target?

Choose 1.0

Reach everywhere

Only if you must run in a browser or a locked-down legacy stack. Accept the ceremony.

Choose 2.0

The sane baseline

Functions, regex, real grouping and types. A huge step up for little cost — a common integration/ESB default.

Choose 3.0

Modern default

Maps, JSON, streaming, higher-order functions. Pick this for new work if your processor (Saxon) supports it.

★ Practical use case — the XML ↔ JSON bridge

Legacy systems speak XML; modern APIs speak JSON. XSLT 3.0 sits in the middle: parse-json() to ingest a JSON payload, transform it, and method="json" to emit JSON — or the reverse, as above. One declarative stylesheet replaces a pile of hand-written mapping code, and it's auditable.
