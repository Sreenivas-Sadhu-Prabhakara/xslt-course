Time to build ciphers — in pure XSLT, with the recursion and byte math from the transformation track. Three of them: a Caesar shift, Base64, and repeating-key XOR. They're superb for learning and useless for security; the point is to understand how these operations work, not to hide data. Folders: examples/08-cipher-caesar, 09-cipher-base64, 10-cipher-xor.

⚠ Read before running

Everything on this page is a teaching cipher. A Caesar shift is broken by hand; fixed-key XOR falls to basic frequency analysis; Base64 hides nothing at all. Do not protect real data with any of them. (See Lesson 9.)

1 · Caesar / ROT13 — recursion over characters

Shift each letter forward by n places, wrapping Z→A, and pass everything else through. It's the string-reverse pattern from Lesson 7, one character at a time. Decryption is the same walk with 26 − n — so ROT13 (n = 13) is its own inverse.

caesar.xsl · the core shift (XSLT 1.0)
<xsl:variable name="UP2">ABC…XYZABC…XYZ</xsl:variable>  <!-- doubled = free wrap -->

<xsl:template name="shift-char">
  <xsl:param name="ch"/> <xsl:param name="n"/>
  <xsl:choose>
    <xsl:when test="contains($UP, $ch)">
      <xsl:value-of select="substring($UP2,
          string-length(substring-before($UP, $ch)) + 1 + $n, 1)"/>
    </xsl:when>
    <!-- lowercase branch, then: -->
    <xsl:otherwise><xsl:value-of select="$ch"/></xsl:otherwise>
  </xsl:choose>
</xsl:template>
$ xsltproc --stringparam shift 3 08-cipher-caesar/caesar.xsl 08-cipher-caesar/message.xml
# plaintext: HELLO, XSLT World! Attack at DAWN. # shift 3: KHOOR, AVOW Zruog! Dwwdfn dw GDZQ. # ROT13: URYYB, KFYG Jbeyq! Nggnpx ng QNJA. # decrypt (shift 3, decrypt=yes): HELLO, XSLT World! Attack at DAWN.

2 · Base64 — encoding, built by hand

Base64 maps every 3 bytes to 4 printable characters. Building it teaches the byte math that 1.0 can't do — so this one is XSLT 3.0, using string-to-codepoints(). Remember: this is encoding, freely reversible by anyone.

base64.xsl · 3 bytes → 4 chars (XSLT 3.0)
<xsl:variable name="i0" select="$b0 idiv 4"/>
<xsl:variable name="i1" select="($b0 mod 4)*16 + $b1 idiv 16"/>
<xsl:variable name="i2" select="($b1 mod 16)*4 + $b2 idiv 64"/>
<xsl:variable name="i3" select="$b2 mod 64"/>
<!-- look each index up in the 64-char alphabet; pad short groups with '=' -->
$ java -jar lib/Saxon-HE-10.9.jar -xsl:09-cipher-base64/base64.xsl -s:09-cipher-base64/message.xml
# encode "Hello, XSLT!": SGVsbG8sIFhTTFQh # matches the system tool: $ printf 'Hello, XSLT!' | base64 → SGVsbG8sIFhTTFQh # decode (decode=yes) round-trips back to: Hello, XSLT!

3 · XOR — where real ciphers begin

XOR each byte of the message with a repeating key. It's the atom inside every stream cipher — and encrypt and decrypt are the identical operation (XOR twice with the same key returns the original). XPath has no bitwise operators, so we compute XOR one bit at a time, which is a fine way to see what XOR actually is.

xor.xsl · bitwise XOR without bit operators
<xsl:function name="c:xor" as="xs:integer">
  <xsl:param name="a"/> <xsl:param name="b"/>
  <!-- result bit i is 1 exactly when a and b differ at bit i -->
  <xsl:sequence select="sum(
     for $i in 1 to 8 return
       if ((($a idiv $POW[$i]) mod 2) ne (($b idiv $POW[$i]) mod 2))
       then $POW[$i] else 0)"/>
</xsl:function>
$ java -jar lib/Saxon-HE-10.9.jar -xsl:10-cipher-xor/xor.xsl -s:10-cipher-xor/message.xml key=SECRET
# "Meet at midnight." XOR "SECRET" → hex: 1e202626653527652e3b213a3a222b266b # feed the hex back with decrypt=yes → Meet at midnight.

⚑ Why these break (and real ones don't)

Caesar has only 25 keys — try them all. Fixed-key XOR reuses the keystream, so the key leaks to frequency analysis and known-plaintext. Real stream ciphers fix this with a never-repeating keystream from a strong generator and a one-time nonce — exactly what a library like cryptography gives you in Lessons 11–12.

★ Practical use case — the legitimate cousins

The mechanics here are used for real, just not for secrecy. Base64 wraps binary attachments in email and embeds images in data-URIs. ROT13 hides spoilers. Simple character maps power transliteration and leetspeak. When you need to move or reshape bytes safely — not hide them — these are the right tools.
