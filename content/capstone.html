Everything you've learned, in one pipeline. A shop sends a batch of orders to a billing partner. On the way, the orders must be transformed into the partner's invoice format, and the sensitive fields must be encrypted — while ids and totals stay readable so middleware can route and reconcile. The partner then decrypts and renders a report. Folder: examples/capstone.

The architecture

Five stages. Each one is a technique from the course; the colour tells you whose job it is — transform, policy/crypto steering, protected output.

Run the whole thing

$ cd examples/capstone && python3 pipeline.py
1 TRANSFORM -> 2 invoices, batch total USD 244.79 2 POLICY -> flagged 6 sensitive fields (email/pan/cvv) 3 ENCRYPT -> 6 EncryptedData blocks; ids/totals still readable: True 4 DECRYPT -> fields recovered on the far side: True 5 RENDER -> out/5-report.html written

On the wire, the sensitive fields are ciphertext but the invoice stays routable:

out/3-wire.xml (excerpt)
<invoice no="INV-SO-1001">
  <billTo>
    <name>Ada Lovelace</name>                    <!-- readable: not sensitive -->
    <xenc:EncryptedData …>…</xenc:EncryptedData>  <!-- was <email> -->
  </billTo>
  <payment>
    <xenc:EncryptedData …>…</xenc:EncryptedData>  <!-- was <pan> -->
    <xenc:EncryptedData …>…</xenc:EncryptedData>  <!-- was <cvv> -->
  </payment>
  <total currency="USD">139.85</total>           <!-- readable: reconciliation -->
</invoice>

Success criteria

  • Invoice totals are correct and computed by the stylesheet, not hard-coded.
  • On the wire, email, pan and cvv are unreadable; invoice ids and total are readable.
  • The receiver reproduces the original sensitive values exactly after decryption.
  • The rendered report never shows a full card number.
  • All cryptography is done by a library — no hand-rolled cipher anywhere.

Make it your own

Extend · easy

Add a field

Encrypt the billing address too. How many files must change? (One — the policy.)

Extend · medium

Per-recipient keys

Wrap the AES key with the partner's public key and ship it as EncryptedKey.

Extend · medium

Second output

Add a stylesheet that emits the invoices as JSON (XSLT 3.0) for a modern API.

Extend · hard

Integrity

Attach an HMAC over each invoice and verify it before rendering (Lesson 12).

⚠ What makes this “secure” and what doesn't

The design is sound — encrypt only what's sensitive, use authenticated AES-GCM from a library, keep routing data in the clear. The demo is not production: it uses a shared SHA-256(passphrase) key. To ship, replace that with per-recipient key wrapping or a KMS, rotate keys, and disable external-entity loading. The XSLT stays exactly the same — which is the lesson.

★ Why this shape shows up everywhere

“Transform to a partner format, protect the regulated fields, stay routable” is the daily reality of B2B EDI, healthcare messaging, and payment processing. If you can build this pipeline, you can read and extend the real ones.
