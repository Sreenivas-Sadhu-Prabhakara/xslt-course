The for-each from Lesson 2 is fine for small jobs, but real XSLT is built on template rules — the language's signature idea. You write one small rule per kind of node and let the processor walk the tree, choosing the right rule for each node it meets. Folder: examples/03-templates.

Push vs pull

Pull style

You drive

for-each reaches out and pulls nodes in a fixed order you dictate. Simple, but the structure lives in one big template.

Push style

The tree drives

apply-templates hands nodes to the processor, which pushes each one to whichever template matches. Rules stay small and composable.

apply-templates in action

apply-templates.xsl
<xsl:template match="/catalog">
  <xsl:text>CATALOGUE&#10;</xsl:text>
  <xsl:apply-templates select="book"/>   <!-- push each book to its rule -->
</xsl:template>

<xsl:template match="book">
  <xsl:text>&#10;* </xsl:text>
  <xsl:apply-templates select="title"/>
  <xsl:apply-templates select="author"/>
</xsl:template>

<xsl:template match="title">
  <xsl:value-of select="."/><xsl:text>&#10;</xsl:text>
</xsl:template>

<xsl:template match="author">
  <xsl:text>    by </xsl:text><xsl:value-of select="."/><xsl:text>&#10;</xsl:text>
</xsl:template>

Each template minds its own node type. Add a new element to the data and you just add a new rule — no existing template needs to change. That composability is why push style scales to big, messy documents.

Two things that surprise beginners

Built-in template rules

Even an empty stylesheet produces output, because XSLT has built-in rules: the default rule for elements just applies templates to their children, and the default rule for text nodes copies the text. That's why unmatched content sometimes “leaks” through as bare text. When it does, add a rule to handle (or suppress) it.

Modes: same node, different jobs

Sometimes you need to process the same nodes twice — once for a table of contents, once for the body. A mode keeps the two sets of rules apart:

modes
<xsl:apply-templates select="tag" mode="inline"/>
...
<xsl:template match="tag" mode="inline">
  <xsl:if test="position() > 1">, </xsl:if>
  <xsl:value-of select="."/>
</xsl:template>

ⓘ When two rules match

If several templates match a node, the more specific pattern wins (book/title beats title beats *). You can set an explicit priority to break ties. Aim to avoid ambiguity rather than rely on the rules.

★ Practical use case — rendering rich documents

DocBook, DITA, TEI and JATS are huge XML vocabularies for books, manuals and research papers. Their publishing toolchains are essentially large sets of template rules: one rule for <section>, one for <figure>, one for <citation>. Push style is the only sane way to render documents whose structure you can't fully predict.

Your turn

The sample data has a <year> in each book that the stylesheet ignores. Add a rule so each book also prints its year. Which template do you add, and where do you push to it?

Show a solution

Add <xsl:apply-templates select="year"/> inside the book template, then a new rule:

<xsl:template match="year">
  <xsl:text>    (</xsl:text><xsl:value-of select="."/><xsl:text>)&#10;</xsl:text>
</xsl:template>

Notice you didn't touch any other template — that's the point of push style.
