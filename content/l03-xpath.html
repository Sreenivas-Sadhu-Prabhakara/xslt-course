XPath is the small query language XSLT uses to point at nodes. You've already used it — match="/", select="title", count(catalog/book) are all XPath. This lesson makes it deliberate. Folder: examples/02-xpath.

Paths select nodes

A path is read left to right, each step moving down the tree from a context node.

ExpressionSelects
catalog/bookevery book child of catalog
catalog/book/titleevery title, two levels down
//titleevery title anywhere in the document
@categorythe category attribute of the context node (note the @)
. / ..the context node itself / its parent
*any element child

Predicates filter

Square brackets keep only the nodes that match a condition.

predicates
book[@inStock='true']            ← books whose inStock attribute is true
book[price < 55]                 ← books cheaper than 55
book[@category='security'][price < 55]   ← both conditions
book[1]         ← the first book       book[last()]   ← the last book

Functions do the arithmetic

XPath 1.0 ships a compact standard library: count(), sum(), string-length(), substring(), contains(), concat(), translate(), position(), last(), and more. (2.0 and 3.0 add hundreds — Lesson 8.)

See them run

queries.xsl prints the result of several expressions as plain text.

$ xsltproc 02-xpath/queries.xsl data/catalog.xml
All titles (child axis): - Learning XSLT - XSLT 2.0 and XPath 2.0 - XML in a Nutshell - Serious Cryptography - Real-World Cryptography - Cryptonomicon Books that are in stock (predicate [@inStock='true']): - Learning XSLT - XML in a Nutshell - Serious Cryptography - Real-World Cryptography Security books under $55 (compound predicate): - Serious Cryptography ($49.95) Aggregations: total books = 6 sum of prices = 272.82 first author = Michael Fitzgerald last title = Cryptonomicon

⚑ The gotcha that catches everyone

Attributes need the @. book[category='security'] looks for a child element named category (there isn't one) and quietly matches nothing. You almost certainly meant book[@category='security']. When a selection is mysteriously empty, check your @ signs first.

★ Practical use case — validation & routing rules

XPath expressions are the business rules in a lot of software. An integration layer routes a message with order[total > 10000] to the approvals queue; a schema check asserts count(//lineItem) > 0; a config UI shows a field only when //feature[@name='beta']/@enabled = 'true'. Learn XPath once, reuse it in XSLT, XML schemas, and dozens of tools.
