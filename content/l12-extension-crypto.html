The final path is the one production systems reach for most: call a real crypto library from inside the transform, through an extension function. The stylesheet stays declarative and readable; the cryptography lives in vetted code. Every serious XSLT stack supports this. Folder: examples/12-extension-crypto.

The idea

You register a function under your own namespace, then call it from XPath like any built-in. Here c:aesEncrypt, c:sha256, and c:hmac are Python functions (backed by the cryptography library) exposed to the stylesheet.

ext-crypto.xsl · crypto called from XPath
<xsl:stylesheet version="1.0"
    xmlns:xsl="http://www.w3.org/1999/XSL/Transform"
    xmlns:c="urn:course:crypto">    <!-- our extension namespace -->

  <xsl:template match="/">
    <xsl:variable name="ct" select="c:aesEncrypt(string(/note/body))"/>
    <envelope alg="AES-256-GCM">
      <ciphertext><xsl:value-of select="$ct"/></ciphertext>
      <sha256><xsl:value-of select="c:sha256(string(/note/body))"/></sha256>
      <hmac><xsl:value-of select="c:hmac($ct)"/></hmac>
    </envelope>
  </xsl:template>
</xsl:stylesheet>
run.py · registering the functions (Python/lxml)
ns = etree.FunctionNamespace("urn:course:crypto")
ns.prefix = "c"
ns.update({"aesEncrypt": aes_encrypt, "aesDecrypt": aes_decrypt,
           "sha256": sha256, "hmac": hmac})   # backed by `cryptography`

Run it: encrypt, hash, authenticate

Notice the envelope carries an HMAC: the receiver verifies integrity before trusting the ciphertext. Authenticated encryption (AES-GCM) does this inherently; the extra HMAC shown here demonstrates the pattern explicitly.

⚠ Processor caveat

Saxon-HE (the free edition used elsewhere in this course) does not allow arbitrary Java extension functions — that's a Saxon-PE/EE feature. For a runnable, free path we use Python/lxml here. The concept is identical across stacks; only the registration API differs.

⚑ Path 2 vs Path 3 — which to use

Use XML Encryption (Lesson 11) when you need an interoperable, standardised document that another party will decrypt with their own toolkit. Use extension functions (this lesson) when the crypto is internal to your pipeline and you want it inline, flexible, and close to the transform. Both keep the cryptography in a vetted library.

★ Practical use case — tokenising a data feed on the way through

An ETL job transforms a customer export and, in the same pass, replaces each email with c:hmac(email) — a stable pseudonym that lets analysts join records without ever seeing the real address. One stylesheet transforms and de-identifies, with the hashing done by a library, not hand-rolled.

Your turn (thinking)

Why does the example verify the HMAC before decrypting, rather than after?

Show answer

To avoid acting on attacker-controlled data. Checking integrity first (“encrypt-then-MAC”) means you reject tampered ciphertext before feeding it to the decryptor, sidestepping a class of padding/parser attacks. It's the same reason AES-GCM authenticates as part of decryption.
